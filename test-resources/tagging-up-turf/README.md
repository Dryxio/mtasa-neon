# Tagging Up Turf co-op prototype

Native `SWEET1A` playback temporarily restores GTA's original model-slot `1` mapping while the scene owns `CSPLAY`. Neon keeps its 61-bone cutscene-player clump out of the ordinary 37-bone CJ clothes cache, then restores MTA's `TRUTH` mapping during teardown. Manual `/tagupsweet1a` validation completed without a freeze or crash, kept CJ visible from the opening shot, and rendered his facial animation without the former jaw and neck deformation.

This resource is the first vertical slice for Neon's planned story runtime. It recreates the gameplay route of GTA: San Andreas' `SWEET1` mission as an authoritative MTA Lua prototype. Its positions and stage order follow the decompiled `main.scm` mission, while its current actor, camera, and tag behaviors are deliberately replaceable experiments.

The target architecture is documented in [`STORY_RUNTIME.md`](../../STORY_RUNTIME.md). The long-term implementation will keep mission/SCM orchestration server-authoritative while moving reusable GTA tasks, tags, camera behavior, and other engine semantics into generic Neon C++ APIs.

## Play

1. Start the resource: `start tagging-up-turf`.
2. Connect one to three players.
3. Run `/tagup`. The player who starts the mission is the driver/leader.
4. Follow GTA's native translated mission prompts and map markers. Hold fire with the spray can while aiming at each marked tag.

Development commands:

- `/tagupsweet1a` is an explicit alias for the full start. It loads and plays GTA's native `SWEET1A` DAT/CUT/IFP scene, then continues directly into the world intro and mission.
- `/tagupidlewood` starts inside the Greenwood at the real Idlewood arrival gate. The car begins slightly rolled above the road so logs can prove that the scene remains blocked after the first wheel contact and starts only when exact opcode `09D0` reports all four contacts.
- `/tagupdeparture` starts inside the Greenwood at the real Ballas arrival gate, with only the earlier tags completed. It uses the same rolled all-wheels test and preserves the actual camera, leave-car, `SWE1_AV`, natural audio-finish, and `05D2` flow.
- `/tagupballas` starts immediately outside the SCM gang-camera gate with the earlier tags completed, while preserving the real Ballas spawn, chat, camera, approach, follow, timer, and attack flow.
- `/taguppickup` starts directly at the post-rooftop recording scene. It holds the party safely above the destination until each client proves the local collision is streamed, then preserves the `0A0B` directional load, fixed camera, two `09F7` horns, `SWE1_BH`, and cleanup flow.
- `/tagupfinal` starts directly at the Grove Street finale, stages CJ and Sweet from the real arrival position, and preserves the complete camera, dialogue, animation, walk-away, skip, and cleanup flow.
- `/taguplosespray` removes the leader's spraycan only while SWEET1 monitors it, providing an isolated `SWE1SPF` failure test.
- `/tagupskip` advances the current stage (leader only).
- `/tagupabort` aborts and restores every participant's previous state.
- `/taguptrace [on|off]` or `F7` controls the local instruction-trace overlay.
- `/taguptracechat [on|off]` sends each new trace operation to the local chatbox.
- `/taguptracepreview` loads a presentation-only sample sequence for visual QA.

## Instruction trace component

`instruction_feed.lua` provides a generic client-side `TAGUP_TRACE` namespace. It is loaded before `client.lua` and receives presentation-only updates from the mission: toggling it cannot affect stage progression. The overlay is invisible by default and places one low-opacity black field below GTA's stock HUD. It makes the exact primitive or opcode the dominant line, keeps the human-readable title as a readable subtitle, and retains the two most recent primitives with explicit status and execution-domain labels.

The live sequence retains full presentation copy and changing runtime evidence for every observed operation even though the minimal overlay does not draw those longer fields. It labels each execution domain honestly as a native cutscene, task, camera, audio, predicate, recording or gameplay primitive, SCM flow, synchronized adapter, co-op barrier, or server-authoritative action. Fast synchronous calls update the card immediately; the trace never delays gameplay for presentation. `/taguptracepreview` remains a purely local visual sample.

`/taguptracechat` is an independent alternative to the HUD feed. When enabled, it writes a color-coded chatbox block for every newly active operation: the exact opcode and complete SCM command in gold, the concrete original GTA `CTask` class in blue when the operation creates one, then the human title in white. Non-task camera, audio, streaming, predicate, and server operations stay on one line rather than being mislabeled as tasks. It deliberately ignores progress-only updates, so tag percentages and per-frame observations cannot flood chat. Enabling it during an active sequence immediately prints the current operation; the chatbox itself provides the durable history.

The feed uses MTA's bundled `pricedown` face for prominent primitives and `bankgothic` for supporting text, so it has no cross-resource font dependency. GTA-style gold, off-white text, green/red/amber history, a heavy black outline, and the restrained background remain readable over bright or detailed scenery without adding a live indicator, counter, or progress bar.

```lua
TAGUP_TRACE.setSequence(steps, {title = "TAGGING UP TURF", subtitle = "CO-OP TRACE"})
TAGUP_TRACE.setCurrent(stepId [, detail])
TAGUP_TRACE.skipTo(stepId [, detail])
TAGUP_TRACE.setStatus(stepId, "queued" | "active" | "done" | "failed" | "skipped" [, detail])
TAGUP_TRACE.fail(stepId [, detail])
TAGUP_TRACE.setProgress(stepId, progress [, detail]) -- progress accepts 0..1 or 0..100
TAGUP_TRACE.reset()
TAGUP_TRACE.toggle([visible])
TAGUP_TRACE.toggleChat([enabled])
```

Sequence entries may be strings or tables containing `id`, `title`, `explanation`, `category`, `primitive`, optional `originalTask`, `detail`, `status`, and `progress`. `explanation` is stable presentation copy; `detail` is the changing runtime evidence supplied by mission hooks. The component only draws local DX primitives or calls the local `outputChatBox`; it owns no timers, elements, mission tokens, or server events. The detailed native-task feed is intentionally available on the mission leader/syncer, which is the client that actually executes and observes Sweet's GTA tasks; other party members do not fabricate those observations locally.

## Current scope

The prototype includes GTA's native `SWEET1A` file cutscene, the following world intro camera, shared co-op objectives, Sweet and his Greenwood, five sprayable tags, a Sweet demonstration, synchronized Ballas combat, mission failure, the return drive, rewards, and restoration of each player's pre-mission state. It uses dimension 4101 so it can run beside other test resources. Its former always-on DX title/objective banner, tag percentages, spray hints, skip hints, and transient French status messages have been removed. Runtime instructions, help, dialogue subtitles, and mission passed/failed banners now use GTA's native GXT block and HUD queues with the original `SWEET1` keys and timings. The F7 instruction trace remains a separate, disabled-by-default diagnostic overlay.

The full start now mirrors the installed `main.scm` boundary before synchronized mission entities exist. Every participant acquires a resource-owned native file-cutscene lease, loads `SWEET1A`, starts only after the server readiness barrier, and reports native camera-spline completion. GTA's own local skip consumption is suppressed for this managed playback; only the leader's original native skip input is sent to the server and broadcast to the party. Every client performs the SCM's immediate black fade and deletes native cutscene data before the server creates Sweet, the Greenwood, and the tags. Abort, failure, disconnect, resource stop, camera takeover, timeout, and stage replacement all use the same native cleanup path. The world intro preserves the paired 8000 ms look-at tasks during `SWE1_AA`, then queues opcode `0639`'s native leader-to-Sweet body turn after that line finishes and immediately starts the 13-second vector camera, without waiting for the turn to complete. Manual single-player validation completed the full native playback at the expected speed, showed the animated spray prop, released the cutscene lease, and continued through `SWE1_AA` to `SWE1_AE` with camera and audio cleanup acknowledged. The leader's native body turn was also accepted after the natural end of `SWE1_AA` and visually matched the intended CJ-to-Sweet rotation. Multi-participant playback remains pending.

Navigation now follows the SCM's blip lifecycle: Sweet's Greenwood has an attached friendly blue blip whenever the player must return to it, driving stages replace it with the default cream destination blip, and every active tag has its exact green coordinate blip until that tag reaches 100 percent. The rooftop tag stays hidden until the player reaches the original approach point.

The current unvalidated property and transition-audio slice restores several mission-visible Greenwood and Sweet properties from `SWEET1`. The Greenwood uses the Greenwoo `0.5 m` base-Z conversion, palette colors `59/34`, plate `GROVE4L`, Bounce FM, a tyre-only no-burst policy, and raw door mode `LOCKOUT_PLAYER_ONLY` during the two SCM lock windows. Sweet receives the verified never-targeted, no-critical-hit, non-drag-out, stay-when-jacked, and upside-down-car flags through a persistent client policy. During the Ballas and rooftop section, the leader reports the phase-specific SCM predicates so the server can freeze Sweet and the Greenwood below the world: Ballas requires both an outside `30/30/10` box and off-screen state, while rooftop accepts either an outside `100/100/10` box or off-screen state. Recording `207` releases that storage before native playback. The transition lines `SWE1_CB`, `SWE1_AS`, and `SWEX_AH`, plus the rotating `SWE1_BM`, `SWEX_BS`, `SWEX_BP`, and `SWE1_BG` return-to-car reminders, now use the native mission-audio pool. Every line loads on each participant behind a server readiness barrier and reports natural completion. `SWE1_AS` blocks the Ballas driving stage exactly as the SCM does, while the other transition lines remain non-blocking. In-game validation of this slice remains pending.

Tag hits now come from Neon's resource-owned native gang-tag service. GTA's real spray-can shot path selects the object, advances its logical alpha by the original increment of 8, updates the native Grove material, returns the original completion state at 255, and emits `onClientObjectGangTagProgress`. The server validates the reporting player or Sweet syncer, weapon, mission stage, object, distance, and exact native delta, then mirrors one authoritative alpha step to the party. It deliberately does not impose a wall-clock delay between valid reports because `CShotInfo::Update` can process several spray shots in one frame. Lua no longer polls fire input, guesses a nearby tag, or advances Sweet's demonstration with a timer. Stream-out, object recreation, resource stop, and conflicting ownership are handled by the generic C++ service while MTA's unsafe global `CTagManager` remains disabled.

The SCM bytecode itself is not executed. Lua keeps the server-authoritative control flow, while verified native services now provide the reusable GTA camera, mission-text, mission-audio, task, animation, and recorded-car behavior used by the implemented slices. This resource is a regression harness and architecture probe, not the intended mission-by-mission implementation strategy.

Sweet's first Idlewood demonstration now exercises Neon's native `setPedExitVehicle`, `setPedTaskSequence`, `getPedTaskSequenceProgress`, and `setPedEnterVehicle` paths using the `SWEET1` profiles. Arrival uses the SCM's 4-by-4-by-4 locate box and exact client-syncer `09D0` all-four-contact predicate rather than the former 11 m sphere and broad MTA ground approximation, and its coordinate blip uses GTA's default cream destination colour rather than the friendly blue reserved for Sweet's car. GTA normally queues `SWEET1_CA` then `SWEET1_AR` immediately before fixed-camera staging; the co-op harness preserves that order but reserves the same two native handles as soon as the party enters the Greenwood. This compatibility prefetch lets unusually slow yet structurally valid replacement banks perform their cold disk load during the drive, then transfers the handles unchanged into the arrival scene. While the resource polls both handles at the barrier, Neon periodically re-arms a native hardware request that GTA accepted into its mission-audio link but silently failed to queue. Silent tracks still follow the normal loaded/play/finished lifecycle, and the server starts only after every client reports both native handles loaded. Release also resets GTA's retained event fingerprint after cancelling the sample, so repeated server reconnects cannot strand all four native slots as apparently foreign ownership. On the driver/syncer, that generic lease sets GTA's verified `bPlayerSafe` pad bit, making the Greenwood receive zero throttle, full brake, handbrake, and the native `0.28` speed clamp instead of a mission-specific velocity reset or freeze. The scene reproduces the three fixed shots, black-screen actor staging, native fades, widescreen, control inhibition, and simultaneous 10-second position/target tracks from the SCM. Sweet receives his native leave-car task first; 600 ms later each player requests the same native task locally, while the authoritative position fallback runs only after the fade is fully black. Only the leader may request a skip, but the server applies that decision to the whole party and restores gameplay under the black fade before every client fades back in.

The vehicle exit is no longer a server-side removal: the leader/syncer requests MTA's authoritative exit lifecycle, observes `TASK_COMPLEX_LEAVE_CAR`, and the server independently confirms `onVehicleExit` before it advances. After black-screen staging, one GTA `CTaskComplexUseSequence` owns the complete `leave_car -> walk -> shoot_at` template. The client observes native task indices `1` and `2` instead of separately assigning and polling the walk and gun tasks. Sweet applies shooting rate `100` and accuracy `90` before dispatch. His mission-actor policy suspends MTA's ordinary player-weapon target substitution, which otherwise replaces the task's tag coordinate because MTA represents script peds as `CPlayerPed` instances. Reaching 100% interrupts the whole sequence and waits the SCM's following 1000 ms.
